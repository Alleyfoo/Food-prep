## Known issues / limitations
- `pairings.ingredient_id` is the FILLER, not the target; cabbage pairings are
  reached via `works_best_with_transformation_id` (this is how `scout()` and the
  round-6 Cook/Scout test resolve them). A future refactor could add an
  explicit `target_ingredient_id` column for clarity, but the FK link works.
- `_detect_subject` is substring-based (like `_detect_ingredient`); very short
  filler names could in principle over-match, but longest-wins + the
  full-ingredient branch gate keep common prompts correct.
- 22 non-cabbage pairings still resolve to corpus 0 (olive_oil, pasta_water, …)
  — `CANONICAL_TO_CORPUS` can be extended.
- `risks` is surfaced in branch/component renders but not yet in the Streamlit
  UI (no UI exists yet).
