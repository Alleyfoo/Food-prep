## Streamlit Cloud framing fix (this session, after the empty-canvas fix)
After `d49ed51` the Cloud app rendered but switching the Ingredient selectbox
(cabbage → broccoli) dropped the card framing and the layout shifted. Root
cause: `st.set_page_config` (layout="wide") and the design CSS `<style>`
injection sat at MODULE TOP LEVEL, so they ran only once per process (first
import). A new session or a later rerun found the module cached in
`sys.modules` → page config reverted to defaults and the CSS was not
re-injected → framing vanished. Locally the CSS survived reruns (Streamlit
1.50 preserves it), masking the bug; Cloud's newer Streamlit does not, so it
only showed there. Fix (`8c1183f`): moved both into `main()` —
`set_page_config` guarded by `st.session_state` (once-per-session, stays the
first st command), CSS `<style>` re-injected every run (idempotent). Verified:
two sessions × cabbage→broccoli, styles + wide layout + card count all
persist, no exceptions, clean log. 98 tests pass.

## Streamlit Cloud fix (this session, post round 12)
After round 12 the deployed Cloud app rendered once then went blank ("No
spinner, empty canvas"). Two stacked bugs, both fixed in `d49ed51`:
1. **Empty canvas on rerun** — `app.py` used `from foodprep.ui.streamlit_app
   import *`, so the module-level rendering (`topbar()` / `st.tabs(...)` /
   `with tabN: ...`) ran only ONCE per process (Python caches the module in
   `sys.modules`). First session rendered; every later session saw an empty
   canvas. Confirmed locally: SESSION-1 = 241k-char page, SESSION-2 = 3.4k
   empty shell. Fix: wrapped the rendering in `main()` and changed `app.py`
   to `from foodprep.ui.streamlit_app import main; main()` — the standard
   Streamlit pattern, runs on every rerun.
2. **SQLite thread error** (masked by bug 1 — cached-module sessions never
   rendered, so the error never surfaced). `CONN` is a `@st.cache_resource`
   connection reused across Streamlit's worker threads; sqlite3's default
   `check_same_thread=True` raised "SQLite objects created in a thread can
   only be used in that same thread" on reruns. Fix: `get_conn()` opens the
   in-memory connection with `check_same_thread=False` (safe — db is built
   once from YAML and only read; Streamlit serializes script runs). Added
   `import sqlite3`; dropped the now-unused `from foodprep.db import connect`.

Verified: fresh server, three sessions all render, tab-click rerun survives,
server log clean, 98 tests pass, all five tab screenshots re-captured +
PIL-verified.

### New module `src/foodprep/export.py`
Two layers (mirrors the plan's "same truth, different surface" rule):
- **Pure renderers** (no `conn`, used by UI + tests):
  `render_branch_markdown(card, partition=None)`,
  `render_component_markdown(d, partition=None)`,
  `render_plate_markdown(r)`, `render_scout_markdown(rows)`.
- **High-level helpers** (take `conn`, used by CLI): `branch_markdown`,
  `component_markdown`, `plate_markdown`, `scout_markdown`. Return `None`
  on a miss (branch/component) so the CLI prints a clean not-found.
- `SCOUT_DISCLAIMER` (verbatim from the UI Scout tab) + `FOOTER_LINES`
  ("## Notes — Generated from local Ingredient Foundry data…") constants.
- **Suggested next move is mechanical** (user-confirmed): a templated line
  read only from `available_now` + `covered_roles` + gaps — never synthesized.
- Shared helpers `_add_section`, `_partition_blocks`, `_add_tags/risks/missing/
  uses/next_moves` keep branch & component renders DRY. The partition render
  (Available now / Missing but useful / No match) is shared by branch, component,
  and plate via `_partition_blocks`.
- Plate render always shows "Missing — hard gaps" + "May want more", then the
  partition (when present), then Risks/Avoid (leans_heavy/dry), then
  "No balance profile for", then "Suggested next move", then the footer.
